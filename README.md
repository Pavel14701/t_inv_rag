# 📘 **Техническое задание: Система алгоритмического трейдинга с DSL, ML‑оценкой вероятности и жёстким риск‑менеджментом**

## **1. Цель проекта**
Создать модульную систему алгоритмического трейдинга, в которой:

- **логика сигналов** описывается декларативным DSL,  
- **индикаторы** рассчитываются высокопроизводительными NumPy/Numba/Polars‑ядрами,  
- **ML‑модель** оценивает вероятность успеха сделки,  
- **риск‑менеджмент** реализован строго в коде и не зависит от LLM,  
- **LLM/RAG** используется только для генерации/модификации стратегий и анализа результатов,  
- **решения о входе/выходе** принимаются детерминированным кодом.

---

# 2. **Компоненты системы**

## **2.1. DSL для стратегий**
Создать декларативный язык (YAML/JSON или текстовый DSL), описывающий:

### **2.1.1. Индикаторы**
```yaml
indicators:
  rsi:
    type: "RSI"
    source: "close"
    length: 14
  ott:
    type: "OTT"
    length: 2
    percent: 1.4
    ma_type: "VAR"
  atr:
    type: "ATR"
    length: 14
    mamode: "rma"
```

### **2.1.2. Условия входа/выхода**
```yaml
entry:
  long:
    - "rsi.value < 30"
    - "price > ott.ott"
    - "trend.direction == 1"
exit:
  long:
    - "price < ott.ott"
    - "rsi.value > 60"
```

### **2.1.3. Параметры сделки**
```yaml
risk:
  stop_loss:
    type: "atr_multiple"
    multiple: 2.0
  take_profit:
    rr: 3.0
```

### **2.1.4. Требования к DSL**
- Парсер должен преобразовывать DSL → AST.  
- AST должен поддерживать:
  - сравнения (`<`, `>`, `<=`, `>=`, `==`)  
  - логические операции (`and`, `or`, `not`)  
  - доступ к индикаторам (`rsi.value`, `ott.ott`, `atr.value`)  
- DSL не должен содержать исполняемого кода.

## **2.2. Индикаторный слой**
Использовать существующие высокопроизводительные реализации:

- RSI, SCRSI, RSI Clouds  
- EMA, SMA, WMA, RMA, HMA, KAMA  
- OTT  
- True Range, ATR  
- ADX, SuperTrend (при необходимости)

Требования:

- NumPy + Numba  
- Polars‑интеграция  
- nan_policy, trim, offset, fillna  
- отсутствие pandas внутри вычислений  

---

## **2.3. ML‑модель вероятности сделки**
Создать модель, которая:

### **Входы (фичи):**
- значения индикаторов на текущей свече  
- производные фичи (RR, стоп, тейк, ATR, волатильность)  
- структура рынка (тренд, импульс, волатильность, фазы)  
- параметры стратегии  
- контекст (время, инструмент)

### **Выходы:**
- вероятность успеха сделки \(P(\text{win})\)  
- опционально: ожидаемое R, вероятность стопа, вероятность тейка  

### **Требования:**
- модель не принимает решений  
- модель не имеет доступа к риск‑менеджменту  
- модель обучается на исторических данных (бэктесты)

Рекомендуемые модели:
- CatBoost / LightGBM  
- MLP / TabNet / FT‑Transformer  

LLM **не используется** для вероятностей.

---

## **2.4. Риск‑менеджмент (жёстко в коде)**
Риск‑модуль должен быть детерминированным и независимым от LLM.

### **Функции:**
- ограничение риска на сделку (например, ≤ 1–2% депо)  
- ограничение дневной/недельной просадки  
- ограничение количества позиций  
- обязательный стоп‑лосс  
- обязательный тейк‑профит  
- фильтры по волатильности  
- фильтры по времени (не торговать в новости)  
- white‑list инструментов  

### **Решение о входе:**
```
если (есть сигнал из DSL)
и (P(win) >= threshold)
и (риск <= лимитов)
→ разрешить сделку
иначе → запретить
```

LLM не может менять лимиты.

---

## **2.5. RAG‑система**
Используется **только** для:

- генерации стратегий в DSL  
- модификации стратегий  
- анализа результатов бэктестов  
- объяснения поведения стратегии  

### **Документы в RAG:**
- спецификация DSL  
- примеры стратегий  
- документация индикаторов  
- результаты бэктестов  
- гайды по интерпретации сигналов  

### **Документы, которые НЕ должны попадать в RAG:**
- риск‑менеджмент  
- лимиты  
- правила открытия позиций  
- параметры исполнения  

---

# 3. **Архитектура системы**

```
[Данные] → [Индикаторы] → [DSL → AST → сигналы] → [ML-модель → вероятность]
                                                ↓
                                       [Risk Engine → решение]
                                                ↓
                                         [Исполнение]
```

LLM/RAG работает **параллельно**, а не внутри контура принятия решений.

---

# 4. **Требования к качеству и производительности**
- Все вычисления индикаторов — NumPy/Numba/Polars.  
- Никаких pandas в вычислительном контуре.  
- ML‑модель должна работать < 1 мс на сделку.  
- Risk Engine — чистый Python/NumPy, без LLM.  
- DSL‑парсер должен работать детерминированно и быстро.

---

# 5. **Ограничения**
- LLM не принимает торговых решений.  
- LLM не меняет риск‑параметры.  
- LLM не генерирует код исполнения сделок.  
- Все решения принимаются только кодом.
